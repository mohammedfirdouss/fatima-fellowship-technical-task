# Qwen3.5-4B-Base Blind Spots Dataset

A probe set for the failure modes ("blind spots") of the base language model
[`Qwen/Qwen3.5-4B-Base`](https://huggingface.co/Qwen/Qwen3.5-4B-Base).

The dataset contains **84 prompts across 12 reasoning categories**:

- **60 failure probes** - 5 per category - hard items the base model is expected to struggle with.
- **24 success controls** - 2 per category - easy items in the *same* domain, used as a baseline.

The controls are the point: with them we can report a **failure rate** ("X% of hard
reasoning probes fail vs Y% of easy controls") instead of a curated pile of anecdotes.
Each record also separates **whether the output was coherent** from **whether the
answer was correct**, so a failure can be attributed to *format* or *reasoning*.

_Model released February 2026 · dataset revision June 2026._

## Dataset Fields

| Field | Type | Description |
|-------|------|-------------|
| `id` | string | Identifier, `{category}_{f\|c}{n}` (e.g. `arith_f3`, `wino_c1`) |
| `category` | string | One of 12 reasoning categories |
| `probe_type` | string | `failure` (hard probe) or `control` (easy baseline item) |
| `prompt` | string | Exact text fed to the model |
| `expected_output` | string | Correct answer |
| `explanation` | string | Why this probe is hard / what correct reasoning requires |
| `runs` | object | Per-dtype results: `runs.float16` and `runs.bfloat16` (see below) |
| `notes` | string | Optional caveats (e.g. dtype provenance) |

Each `runs.<dtype>` object holds one generation:

| Sub-field | Type | Description |
|-------|------|-------------|
| `model_output` | string | Raw text generated by the model under that dtype |
| `output_coherent` | bool | Was the output legible, on-format text (vs symbol-salad)? |
| `answer_correct` | bool | Given a coherent output, was the answer right? |
| `failure_mode` | string | `format_failure`, `reasoning_failure`, `correct`, or `pending` |

The `runs` object has slots for both `float16` and `bfloat16`. The 12 original
prompts carry `runs.float16` results (all incoherent - see the caveat below); the
remaining records carry `runs.bfloat16` from the primary bfloat16 run.

`dataset/data/prompts.jsonl` holds just the inputs (no outputs) and is the canonical
source the notebook and Modal runner consume. `dataset/data/train.jsonl` is the full
record set including both dtype runs and their labels.

## Methodology: Two Axes, Three Outcomes

Per a key piece of review feedback, we **do not collapse "wrong" and "incoherent"
into one bucket**. Every output is coded on two independent axes:

1. **Coherence** - did the model produce legible, on-format English? A base model that
   emits punctuation/Unicode salad has not even entered the Q&A game.
2. **Correctness** - *conditional on coherence*, is the answer right?

This yields three outcomes:

| `failure_mode` | Coherent? | Correct? | Interpretation |
|---|---|---|---|
| `format_failure` | No | - | Model never produced a usable answer (doesn't follow the QA format) |
| `reasoning_failure` | Yes | No | Model understood the task but mis-reasoned |
| `correct` | Yes | Yes | Success (the baseline we measure controls against) |

The distinction matters: **format failure** is an alignment/format problem fixable with
light instruction tuning, whereas **reasoning failure** is a capability gap requiring
chain-of-thought or domain data. Conflating them would point at the wrong fix.

Coherence and correctness are first auto-labelled by simple heuristics (a printable-character
ratio for coherence; normalized containment of the expected answer for correctness) and
then **spot-checked by hand** - the heuristics are a screen, not a judge. See `classify()`
in `modal_runner.py` / the notebook.

## ⚠️ Important Caveat: float16 vs bfloat16

The first batch of captured outputs (the 12 records currently carrying a `notes` field,
generated under commit `dc58516` which set `dtype=float16`) are **incoherent symbol-salad
across every category** - including the controls and even items the model clearly "knows".
One probe (`logic_f1`) even begins with the correct token "No," before collapsing into noise.

Uniform, content-independent garbage like this is the classic signature of **float16
numerical overflow on Qwen-class models**, *not* a genuine reasoning blind spot. float16's
narrow exponent range can overflow in attention/MoE layers; bfloat16 has the range to stay
stable.

**Therefore those 12 `format_failure` labels are confounded and should be treated as
preliminary.** The float16 results are preserved in `runs.float16` for reference. The
primary analysis uses **bfloat16**. The Colab notebook runs 12 prompts (1 failure per
category) for speed on a T4; the full 84-prompt set is in `prompts.jsonl`. The Modal
runner supports both bfloat16 and float16 for completeness.

## What Fine-Tuning Dataset Would Fix These Errors?

These errors stem from at least three distinct root causes, and fixing them likely
requires separate fine-tuning interventions. **Which fix applies depends on the
format-vs-reasoning split above**: pervasive format failure on a base model is largely a
*formatting/instruction-following* problem (the model can't be coaxed into the QA frame),
which light instruction tuning addresses, whereas reasoning failures need the
capability-focused data below.

### 1. Reasoning / Chain-of-Thought Data

Categories affected: arithmetic, cognitive_reflection, spatial_reasoning,
temporal_arithmetic, probabilistic_reasoning, multi_hop_ordering, formal_logic.

The model needs training examples that demonstrate **step-by-step intermediate
reasoning** rather than jumping to a final answer. High-quality datasets:

- **OpenMathInstruct-2** - math word problems with detailed solutions
- **NuminaMath-CoT** - competition mathematics with chain-of-thought
- **LogiQA / ReClor** - formal logical reasoning
- **ProntoQA** - synthetic proof-of-concept logical entailment chains
- Synthetic step-by-step reasoning traces distilled from a capable teacher model

### 2. Factual Correction / Misconception Debunking Data

Categories affected: factual_misconception, linguistic_illusion.

The model requires exposure to authoritative corrections of common myths, alongside
the false claims themselves so it learns to contrast and reject them. Sources:

- **FactCheck** and **Snopes** articles converted into structured Q&A pairs
- Wikipedia "common misconceptions" page parsed into correction triples
- Synthetic contrast pairs: (false claim, "This is false because...")
- False-premise / presupposition examples to teach the model to reject the frame

### 3. Character-Level and Token-Agnostic Tasks

Categories affected: character_level_reasoning, winograd_schema.

Tokenization causes the model to "see" words as chunks rather than characters.
Fine-tuning on explicit character-level and entity-level tasks can partially mitigate this:

- Synthetic character-counting/spelling/reversal examples across diverse vocabularies
- **Winogrande** - large-scale Winograd schemas for coreference

### 4. Commonsense Physics

Categories affected: commonsense_physics.

The model needs training where the physically correct answer diverges from naive
arithmetic. Sources: **PIQA**, plus authored counterintuitive physics vignettes with
explanations.

## How Big a Dataset Would Be Needed?

| Error Type | Estimated Examples | Rationale |
|---|---|---|
| Chain-of-thought reasoning | 10,000–50,000 | Broad skill transfer; a few thousand quality CoT traces meaningfully help a 4B model (Magister et al., 2023) |
| Factual / misconception correction | 2,000–5,000 | Targeted belief updates from quality contrast pairs |
| Character-level tasks | 5,000–20,000 | Many examples needed to override tokenization bias |
| Commonsense physics | 3,000–10,000 | Moderate diversity to cover physical scenarios |

A combined **~20,000–50,000 high-quality examples** (quality over quantity) is a
reasonable starting point for a 4B model, plus a small instruction-format set if the
bfloat16 run confirms residual format failures. For the math/logic reasoning failures,
distillation from a larger teacher has been particularly effective for 1–7B models.

## How to Assemble Such a Dataset

1. **Existing benchmarks** - reformat LogiQA, Winogrande, PIQA, OpenMathInstruct, NuminaMath.
2. **Synthetic generation** - teacher-model step-by-step traces, filtered by correctness.
3. **Myth-correction scraping** - Snopes / Wikipedia misconceptions → (claim, correction, source) triples.
4. **Programmatic generation** - scripts emit unlimited character-counting, modular-arithmetic,
   and transitive-ordering examples with guaranteed-correct labels.
5. **Human annotation** - a small annotator pool reviewing synthetic drafts for the subtler
   commonsense-physics cases.

## Reproducing

The accompanying repo provides a Colab notebook and a Modal runner. Both read
`dataset/data/prompts.jsonl` and run all 84 prompts with greedy decoding, auto-classify
each output, and print the failure-rate summary. The Colab notebook runs **bfloat16
only**; the Modal runner supports both dtypes.
